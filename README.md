# Natural Language-to-SQL Agentic Pipeline

A from-scratch end-to-end agentic NL→SQL pipeline built with `LangChain`, `Claude`, and `Voyage AI` — evaluated on the BIRD Mini-Dev benchmark. The project is structured as follows:
* **Stage A**: Core agent on Chinook — clean and small, good for learning the fundamentals without BIRD's real-world messiness getting in the way
* **Stage B**: Self-correction loop (still on Chinook)
* **Stage C**: Eval harness, using BIRD — pick a handful of its (question, gold SQL) pairs, measure accuracy
* **Stage D**: Schema retrieval — pick one of BIRD's larger/messier databases, show accuracy with vs. without retrieval
* **Stage E**: Gradio app → deployed to [Hugging Face Spaces](https://huggingface.co/spaces/anantshri1/nl2sql-agentic-pipeline)

---
## Manual NL→SQL chain

### Core Agent on [Chinook Database](https://github.com/lerocha/chinook-database)

Dependencies installed: `langchain, langchain-anthropic, langchain-community, sqlalchemy`.

The original plan was to download a prebuilt `Chinook.sqlite` file from a Google Cloud Storage bucket referenced in LangChain's own tutorial, but that asset no longer exists (confirmed via a `NoSuchKey` XML error after inspecting the tiny downloaded file). The fix was to build the database differently and more durably: fetch the actual SQL script (`CREATE TABLE` and `INSERT` statements as plain text) from the actively maintained `lerocha/chinook-database` GitHub repository, execute it into an in-memory `SQLite` connection via Python's `sqlite3` module, and wrap that connection in a `SQLAlchemy` engine using `StaticPool` with a creator lambda. This was necessary because an in-memory SQLite database is tied to a single connection object — without `StaticPool` forcing reuse of that exact connection, the data would vanish after the first query. The resulting engine was wrapped in LangChain's `SQLDatabase` utility, which exposes the schema (`table names, columns, sample rows`) as a text string via `get_table_info` — this is how the LLM later learns what's in the database.

The first real NL-to-SQL chain was then built using `LangChain` Expression Language: 
* a `ChatPromptTemplate` piped into the LLM (`prompt | llm`), with the database schema and dialect injected into the prompt and the natural-language question as the human message;
* a `clean_sql` helper function was added to strip markdown code fences, since LLMs sometimes wrap SQL in triple-backtick fences despite explicit instructions not to;
* an `ask_db` function combined these: build the schema, invoke the chain, clean the output, execute it via `db.run`, and return the result. 

> Tested successfully on "How many tracks are there in total?", returning a correct count.

### Self-Correction Loop

The correction logic was built and tested in isolation before being wired into a full loop, specifically because Claude at `temperature 0` reliably succeeds on Chinook's small schema on the first attempt — meaning a full retry loop run on normal questions might never actually exercise the correction path. To force a test of the correction path, a deliberately broken query (`SELECT COUNT(*) FROM Tracks`, using the wrong plural table name) plus its real `SQLite error` message were manually fed into a correction chain.

With the correction logic validated, it was wired into `ask_db_with_retry`: generate an initial query, then loop up to `max_retries` times, attempting execution via `db.run` inside a try block; on any exception, the error message is fed into the correction chain to produce a new query for the next attempt. A successful run returns immediately from inside the loop; exhausting all retries raises a RuntimeError rather than silently returning a wrong answer. Tested successfully on Chinook, succeeding on the first attempt with no correction triggered (expected, given Chinook's simplicity).

## BIRD Mini Dev Benchmarking

The existing `ask_db_with_retry` pipeline used `db.run()`, which returns query results as a stringified Python list (e.g. "`[(0.0657...,)]`") rather than structured data. This is fine for feeding results back to an LLM, but useless for programmatic grading, since comparing stringified floats produces false negatives from formatting alone. The fix was a dedicated grading helper, `execute_for_grading(db, sql)`, which calls `SQLDatabase`'s `internal _execute()` method instead of `run()` to get real Python tuples with correct types preserved. This function is used only for grading and never touches the existing agent-facing retry loop, keeping that path unchanged.

On top of that, `is_correct(predicted_sql, gold_sql, db)` was built to do execution-accuracy comparison. A deliberate methodological choice was made here: exact ordered-tuple comparison was used instead of BIRD's standard set-based comparison (which ignores row order and duplicates). This is stricter than the official BIRD metric, meaning the accuracy numbers in this project are a conservative lower bound relative to what BIRD's own leaderboard would report for equivalent SQL. The function treats any predicted SQL that fails to execute as simply incorrect, rather than raising, since a single bad query should never crash a 30-example evaluation run.

`ask_db_with_retry` was also adjusted at this point: previously it returned the executed result string, but the eval loop needs the actual SQL text to regrade it independently, so it was changed to return the SQL string on success while still validating execution internally (to keep the self-correction retry trigger working), and to keep raising `RuntimeError` on total failure after `max_retries`.

With those pieces in place, `run_eval(examples, db, max_retries)` was built as the main evaluation loop: it iterates over a list of BIRD examples, generates a prediction for each via `ask_db_with_retry`, grades it with `is_correct`, and stores a full per-example result dictionary (`question_id, question, predicted_sql, gold_sql, correct`) rather than just a running tally, specifically so failures can be inspected afterward rather than only knowing a final percentage. RuntimeError from total generation failure is caught at this loop level (not inside `ask_db_with_retry` itself), keeping the division of responsibility clean: the generation function stays loud about failure, the eval loop is what knows one bad example shouldn't kill the other 29.

The first full run, on the `debit_card_specializing` database (30 examples, 5 tables: customers, gasstations, products, transactions_1k, yearmonth), produced **43.3% accuracy (13/30 correct)**. Notably, every single example executed successfully on attempt 1 — meaning the self-correction retry loop, while functioning correctly, never actually fired during this run, because all 17 failures were syntactically valid SQL that computed the wrong answer. This is an important finding in itself: retry-on-error can only catch one class of failure (queries that don't execute) and is structurally blind to semantically-wrong-but-valid SQL, which turned out to be the dominant failure mode.
A substantial amount of this conversation was spent manually inspecting failures to build an empirically-grounded failure taxonomy, rather than guessing at categories. A show_failure helper was built to print a failing example's question, evidence, predicted SQL, gold SQL, and both actual result sets side by side. Going through roughly 17-18 failures (the exact count varied slightly between repeated runs due to model non-determinism, discussed below) produced seven distinct, evidence-backed failure categories:

1. Wrong aggregate / didn't follow evidence's formula literally — 1473 (`SUM` vs `AVG`)
2. Sign/semantic logic correct, but tiny floating-point rounding diff — 1476: `402524570.16999084` vs `402524570.1700022` — these differ in the 9th significant digit. This is a new category: floating-point accumulation order differences (`CASE/SUM` vs `IIF/SUM` can sum in different order → tiny float drift). Under strict ordered-tuple equality, this scores as wrong despite being correct to 8+ significant figures.
3. Format/shape mismatch, same underlying answer — 1490 ('`201304`' vs '`04`')
4. Wrong subquery/filter logic entirely, predicted answer structurally diverges — 1498 (gold picks "the min consumption row" customer, not "min per segment, then average" — predicted built a completely different multi-CTE structure than what evidence's wording wanted), 1501 (massive divergence — wrong formula scaffolding), 1505 (wrong WHERE column entirely — predicted used `Consumption > 46.73` against per-row consumption but didn't get denominator right per evidence), 1500 (interpretation difference, established already)
5. Wrong table/column joined — predicted query silently returns empty result — 1514, 1521 (need re-check from earlier list, but 1514 here: predicted filters `Date LIKE '2013-06%'` on `transactions_1k.Date`, but that table's date format apparently isn't `'2013-06%'`-shaped the way `yearmonth.Date` is, or the wrong table is being filtered—gold filters via `yearmonth` join instead, predicted's filter never matches anything, returns `[]`). Same root cause in 1505 (1505 in this batch — September 2013 product list — predicted's `strftime('%Y-%m', t.Date)` also returns `[]`, meaning `transactions_1k.Date` isn't in the format `strftime` expects, or it's the wrong table to filter by date at all — gold filters on `yearmonth.Date` matching customer, not on `transactions_1k.Date` directly).
6. Wrong domain concept — "currency" vs "nationality/country" — 1525: question asks "currency," predicted correctly returns Currency. But 1526 — question asks "nationality," predicted returned Currency ('CZK') instead of Country ('CZE'). That's a real semantic bug: nationality ≠ currency, and the model conflated them.
7. Picked the wrong specific row due to ambiguous referent ("the customer who paid X") — multiple customers/transactions matched same price — 1529: gold's own subquery resolves to a different price (1513.12) than the question states (634.8) — this looks like a likely error in BIRD's own gold SQL/dataset, not a fixable model issue. Gold result is `None`. This is worth flagging as a dataset quirk, not a model failure.
8. Column order swapped vs. gold, otherwise correct values — 1528: predicted returns (`3437.01, 537255.52`), but gold returns (`68740.20, 3437.01`) — predicted computed total spend from `transactions_1k.Price` only for that customer, while gold's total seems much larger — worth flagging as unconfirmed, needs a closer look rather than guessing.
Right concept, wrong "top" entity due to wrong ranking table — 1531 (already covered — ranked by `transactions_1k.Price` instead of `yearmonth.Consumption`)
9. Right values, wrong/extra grouping — gold collapses duplicates predicted doesn't, or vice versa — 1533: predicted returns 9 distinct customer rows; gold returns 10 rows including a duplicate 126157.7 appearing twice. Gold isn't deduplicating per customer-transaction-row; predicted used `DISTINCT` on customer. Same root issue as the unit-of-aggregation ambiguity in 1498 — "give their consumption status" read as "per person" by predicted vs "per qualifying transaction" by gold.

Summarizing these:

| No.|  Category | Detector heuristic|
| ----| ---------|-------------------|
|1| Wrong aggregate function| predicted/gold both succeed, evidence contains an explicit formula string the SQL doesn't match (hard to detect automatically — flag for manual review)|
|2| Floating-point near-miss| both numeric, both same row/col count, abs(p-g)/abs(g) < 1e-6 for all numeric cells → near-miss, not true failure |
|3| Format/shape mismatch, same magnitude| string result differs but one is a substring/reformatting of the other|
|4| Predicted returns empty result `[]` | trivially detectable — any predicted result `== []` while gold isn't|
|5| Wrong table/column for the underlying concept| hard to auto-detect; flag when result entity-IDs differ entirely (e.g. different `CustomerID` in result)|
|6|Row-count mismatch with same row content (just duplicated/deduped differently)|same `set(predicted) == set(gold)` but `sorted(predicted) != sorted(gold)` due to length, or `len` differs by small amount|
|7|Likely gold-SQL dataset error|manual flag only — can't auto-detect|

Of these seven categories, two were determined to be cheaply and reliably automatable from execution output alone: floating-point near-miss (via relative tolerance comparison, using `max(abs(gold), 1e-9)` as a denominator floor to avoid division by zero, scaled relatively rather than with a flat epsilon so it works correctly across results ranging from single digits to hundreds of millions) and empty-predicted-result (a trivial equality check against an empty list). A `categorize_failure(result, db)` function was built covering exactly these two cases, deliberately leaving everything else tagged `needs_manual_review` rather than attempting to auto-detect semantically subtle categories like wrong-table-for-concept or aggregation-unit ambiguity, since distinguishing those reliably requires understanding query meaning, not just shape. This function was wired into `run_eval` so that every failing example is automatically tagged inline during evaluation, with the tag only added when an example is actually incorrect (correct examples carry no `failure_category` key at all).

A significant secondary finding came from re-running the evaluation multiple times: despite running at `temperature=0`, results were not perfectly deterministic between runs. This was initially concerning, but was resolved by running the full evaluation three times via a new `run_eval_multiple(examples, db, n_runs, max_retries)` function, which tracks per-question_id correctness across all runs and reports both the per-run accuracy and a count of how many examples "flip" between correct and incorrect across runs. 
**The result**: accuracy was stable at exactly 43.3% across all three runs (mean 43.3%, range 43.3% to 43.3%), and only 2 of 30 examples (roughly 7%) flipped correctness at all between runs, with the other 28 fully deterministic in outcome even though `temperature` was 0. The two flipping examples were both traceable to *genuine decision-boundary sensitivity*: one flipped due to the model occasionally adding an extra descriptive column to its `SELECT` (correct core answer, but failing strict tuple comparison), and the other flipped due to the model alternating between `COUNT(*)` and `COUNT(DISTINCT)` on a question with genuine aggregation-unit ambiguity. The conclusion drawn was that `temperature=0` controls token-selection determinism but not underlying floating-point/serving-level determinism in LLM inference at scale, and that this is a known, provider-agnostic property of how LLM APIs are served, not a project-specific bug. This finding is positioned as a deliberate, quantified result rather than noise to be hidden: **accuracy is reported as stable at 43.3% plus or minus negligible run-to-run variance, with 93% of examples behaviorally deterministic**.

## Schema Retrieval 

> **Why schema retrieval?** A deliberate discussion took place about what actually could and could not improve model accuracy, given that the underlying LLM (Claude, accessed via API) is a frozen, closed-weight model. The conclusion: every lever available in this project (prompt engineering, evidence formatting, retry loops, schema retrieval) operates on the model's inputs and environment, not on its underlying learned knowledge; only fine-tuning (which would require switching to an open-weight model and was explicitly scoped out as a separate future project, alongside a more complex LangGraph/LangSmith-based extension) can change the model's actual weights. This framing directly motivated treating schema retrieval as the next legitimate accuracy lever, since it is the most sophisticated way to improve what context the model receives, rather than attempting ad-hoc prompt patching tuned to the specific 30-example sample already observed.

The target database was confirmed empirically (not assumed) by querying table counts across all 11 BIRD Mini-Dev databases directly: `formula_1` was confirmed as having the most tables of any database in the set (13, tied for the maximum, versus a next-highest of 10 for `superhero`), with 66 associated questions, making it the intended "larger, messier" database called for in the original project plan. The full schema was inspected directly, revealing concrete, real ambiguity that motivates retrieval as more than a toy exercise: 
* `points` appears in three different tables: `constructorResults.points`, `constructorStandings.points`, `results.points` (and implicitly `driverStandings.points`). A question like *"how many points did X score"* is genuinely ambiguous about which points — race-by-race result points vs. cumulative standings points vs. constructor-level points. Pure keyword match can't disambiguate this at all; even semantic embedding similarity might struggle since all four are legitimately about "points."
* `position` appears in `results`, `driverStandings`, `constructorStandings`, `lapTimes`, `qualifying` — same problem, worse: `"position"` in `lapTimes` means track position during a lap, in `qualifying` means qualifying rank, in `driverStandings` means championship rank. These are semantically very different concepts sharing one column name across 5 tables.
* `raceId` is the join key threading through 9 of the 13 tables — meaning almost any question about "a race" plausibly touches many tables simultaneously, so "relevant tables" for a given question is often genuinely multi-table, not single-table. Retrieval here is more "which subset of 3-4 tables" than "which single table."

**Retrieval Chunk Design**: The retrieval approach settled on was **hybrid**: BM25 keyword search combined with embedding-based semantic search, on the reasoning that schema retrieval has a specific property distinguishing it from generic document RAG — table and column names frequently contain the exact keyword used in a question (favoring BM25), while other questions use different wording for the same underlying concept (favoring embeddings), and this split was already visible empirically in wrong-table-for-concept failures. `Voyage AI` was selected as the embeddings provider, confirmed via direct lookup to offer a free tier of 200 million tokens for its current embedding models, which is effectively unlimited for a schema corpus of this size. The retrieval unit was settled as one chunk per table (full `CREATE TABLE` definition plus sample rows), rather than per-column chunking, since the retrieval decision being made is which tables to include in a prompt, not which individual columns.

Each of the 13 tables was indexed as a single chunk using `LangChain`'s `db.get_table_info(table_names=[t])` called per-table. This was chosen over building a custom schema formatter because SQLite's DDL output already includes `FOREIGN KEY` declarations inline — confirmed empirically by inspecting the `pitStops` table chunk before committing to this approach — meaning foreign-key relationships were available to the retriever for free, with no extra extraction work.

**BM25 Retrieval and Tokenizer Bugs**: Built using `rank_bm25`'s `BM25Okapi` over all 13 table chunks. The `tokenizer` went through two rounds of empirically-discovered bugs:
* first, a naive lowercase-and-strip-punctuation tokenizer failed to split `camelCase` identifiers (`pitStops` never matched a query containing "pit" and "stops" as separate words), discovered by running a real query and manually checking token overlap rather than trusting the output. The fix added a regex-based `camelCase` split (inserting spaces at lowercase-to-uppercase boundaries) and concatenated both the original and split forms.
* second, that same `camelCase`-handling tokenizer was later found to silently self-duplicate any plain-English query with no `camelCase` content (since the "split" version was identical to the original, and both were being concatenated unconditionally), inflating every query token's term frequency and subtly distorting BM25 rankings. The fix only concatenates the split form when it actually differs from the original.

**Voyage AI embeddings**: All 13 table chunks were embedded using `voyage-4` with `input_type="document"`, and queries were embedded separately with `input_type="query"`, per Voyage's asymmetric retrieval design. *An adversarial test query ("highest-attended competition" — sharing zero literal tokens with the races table) confirmed the embedding model could surface races via pure semantic similarity, validating that embeddings were doing real conceptual work rather than disguised keyword matching*. Cosine similarity (not dot product) was used for scoring, normalizing across the differing raw magnitudes of variable-length table chunks. Voyage's free tier (200M tokens) was confirmed more than sufficient for the whole project's scale (a few thousand tokens for the one-time table indexing, more for repeated queries during experimentation), though the account's default 3 RPM rate limit (prior to adding a payment method) required explicit throttling (`time.sleep delays`) in any loop making repeated Voyage calls, to avoid mid-run `RateLimitErrors`.

**Reciprocal Rank Fusion (RRF)**: BM25 and embedding rankings were combined via RRF (`k=60`, the standard default), which fuses based on rank position rather than raw score magnitude, since BM25 scores (unbounded term-frequency weights) and cosine similarities (bounded -1 to 1) aren't directly comparable. Both individual rankings cover all 13 tables; only the final fused list is truncated to the experiment's top-k. A documented edge case: for the adversarial "competition" query, the races table ranked 2nd by embeddings alone but scored a hard zero on BM25 (no token overlap at all), and RRF's fusion formula caused races to drop out of the final top-5 entirely — a concrete, reproducible demonstration that RRF rewards cross-method consensus and can suppress a genuine single-method signal when the other method actively disagrees.

**Ground-truth table extraction (`extract_tables_from_sql`)**: To measure retrieval quality against BIRD's gold SQL, a function was built using the `sqlparse` library to walk each gold query's token tree and collect real table names referenced after `FROM` or `JOIN` keywords. This function went through five rounds of real, empirically-discovered bugs before being trusted: 
1. an initial version only handled bare `JOIN`, missing all `INNER JOIN / LEFT JOIN / etc.` variants, since `sqlparse` emits these as single `Keyword` tokens whose value ends with "`JOIN`" rather than matching the literal string "`JOIN`";
2. the initial version didn't recurse into subqueries at all, missing tables referenced inside a `FROM (SELECT ...)` construct;
3. after adding recursion, an aliased subquery's own alias (e.g. the sub in `FROM (...) AS sub`) was incorrectly added as if it were a real table, because `sqlparse` wraps an aliased subquery as a single `Identifier` token whose `get_real_name()` returns the alias directly;
4. CTE names declared via `WITH name AS (...)` were similarly leaking in as if they were real schema tables;
5. the same `INNER JOIN` keyword-matching gap from issue (1) was independently rediscovered when testing against real BIRD examples, since two of four hand-picked "hard" examples used `INNER JOIN` inside subqueries. 

The final function checks for nested parentheses before trusting an identifier's real name (to avoid alias leakage), separately collects and excludes CTE-declared names, and matches any keyword ending in "`JOIN`" rather than an exact-match list. It was validated against all 66 `formula_1` gold SQL queries with zero empty extractions, a sane distribution of 1-4 tables per question, and manual spot-checks (both a random sample and the four hardest hand-picked examples with subqueries/CTEs) all confirmed correct against the underlying English questions.

**Retrieval-quality metric**: For each of the 66 questions, `hybrid_retrieve()` was run and its top-k table set was compared against the gold table set from `extract_tables_from_sql`. Two metrics were tracked: *recall* (binary — did the retrieved set fully contain the gold set) and *overlap fraction* (the proportion of gold tables actually found, a more granular signal). A k-sweep across k=3, 5, 7 showed retrieval recall climbing steadily and not yet plateauing: 

|k|recall|overlap|
|-|------|-------|
|3|53.0% | 75.6%|
|5| 71.2%| 87.1%|
|7| 83.3%| 93.4%|


* `races` misses `[892, 906, 930, 962, 954]`: this is exactly the named-entity-grounding problem we found for `drivers`, just hitting a different table. "Lewis Hamilton's first race" [906], "Japanese drivers from 2007 to 2009" [954], "1983/7/16" [881] — these reference specific races by date, driver name, or implicit time range, never the literal word "race" in a way that matches races' own sample-row content (which would contain different specific dates/names than whatever the question asks about). Same mechanism as `drivers`, different table — proper nouns and dates don't ground against a 3-row sample.
* `results` misses `[977, 881]` are a genuinely different and more interesting mechanism. Look at [977]: "From race no. 50 to 100, how many finishers have been disqualified?" — this question is using abstract/derived-status vocabulary ("finishers," "disqualified") that maps to the `status` table's content (status values like "`Disqualified`," "`Finished`") rather than to `results`' own column names directly — even though the correct join actually needs `results.statusId` joined against status.status. The retriever (correctly, in a sense!) found status as highly relevant — but the question doesn't surface any vocabulary that screams "results," because "`finishers"/"disqualified`" are status-table concepts, and results is just the bridge table holding the foreign key, with no standalone textual signal of its own that this question needs it. This is a join-bridge invisibility problem: a table that's structurally necessary (it's the only place `raceId + statusId + driverId` come together) but has no semantic fingerprint matching the question's own wording, because the wording describes the destination concept (disqualification status) not the junction table that connects to it. [881] is a softer version of the same thing: "race completion rate" is a derived/computed concept (you'd compute it from results, likely via status or laps vs expected laps) — the word "completion" doesn't appear anywhere in results' DDL or sample data, only the underlying columns that enable computing it do.

We identify three distinct, named failure mechanisms:

|Mechanism|Example|Why retrieval misses it|Potential Fix|
|--------|------|---------------------------|-----------|
|1. BM25 hard-zero suppression|"`competition`" vs races (the original adversarial test)| Vocabulary mismatch; BM25 contributes a zero that RRF can't overcome even when embeddings alone found it| Tune RRF weighting or `k`|
|2. Named-entity grounding gap| "Bruno Senna," "Lewis Hamilton," specific dates|Proper nouns/dates in the question don't match the specific different names/dates in a table's 3-row sample — no amount of semantic matching helps, since this isn't a vocabulary problem|Fundamentally out of scope for schema-only retrieval, needs entity linking|
|3. Join-bridge invisibility|"finishers," "disqualified," "completion rate"|The question's vocabulary matches a different, semantically-adjacent table's content; the actually-needed bridge table has no textual fingerprint of its own|Maybe FK-aware retrieval, where retrieving status should pull in tables it foreign-keys agains|

**Generation pipeline integration**: `ask_db_with_retry` was extended with an optional schema parameter (defaulting to `None`, falling through to the `original db.get_table_info()` behavior, preserving all existing call sites unchanged). Two schema-builder functions were added: `build_full_schema` (a thin wrapper for symmetry, cached once per database since it's question-independent) and `build_retrieved_schema` (rebuilt per question, since retrieval depends on the question text). `run_eval` was similarly extended with an optional `schema_fn` parameter (a function from question to schema string), again defaulting to `None` to preserve original behavior.

> **Reliability issues encountered and fixed during generation runs** A `SQLAlchemy` parameter-binding bug surfaced when a gold SQL query contained a literal question mark inside a `LIKE` pattern string; SQLAlchemy's text-execution layer misinterpreted it as a bind-parameter placeholder, throwing an unhandled exception that crashed the entire 66-example eval loop, since `is_correct` only wrapped the predicted-SQL execution in a try/except, not the gold-SQL side. Fixed by wrapping both sides defensively. Separately, a Colab disconnect mid-run caused total loss of all in-memory results up to that point, since the original `run_eval` had no persistence; checkpointing was added (writing results to a JSON file after every single example, with `files.download()` to pull a copy to the local machine), making subsequent runs resilient to disconnects.

**Full-schema vs. retrieved-schema generation accuracy**: Across all 66 questions: full-schema baseline scored 50.0% (33/66); retrieved-schema (hybrid, k=5) scored 54.5% (36/66). A direct diff of which specific questions flipped in each direction (4 fixed by retrieval, 1 broken by retrieval, 29 failing under both conditions, 32 succeeding under both) showed the net positive result was real but mixed in mechanism: three of the four "fixed" examples showed logically identical SQL between conditions, differing only in minor formulation choices (a missing `DISTINCT`, position vs. rank) that plausibly reflect run-to-run noise rather than retrieval-driven improvement; one fix was a genuine structural win, where full-schema picked the wrong but column-similar results table for a "fastest lap" query while retrieved-schema was forced toward the correct `lapTimes` table because results wasn't offered as an option; the one regression was a clean instance of the join-bridge-invisibility failure mode, where results was excluded from the retrieved top-5 for a "completion percentage" question and the model substituted an incorrect table instead.

**Three-way ablation (BM25-only vs. embedding-only vs. hybrid)**, *all at k=5, on generation accuracy*: BM25-only scored 48.5% (32/66), slightly below the full-schema baseline, consistent with the hard-zero-suppression weakness named earlier; embedding-only scored 53.0% (35/66); hybrid (RRF-fused) scored 54.5% (36/66), the best of the three. This ordering supports the use of hybrid fusion over either method alone, though the margins are narrow enough (a few examples out of 66) that they should be read as suggestive rather than statistically definitive without repeated runs.

**Retrieval-at-k sensitivity sweep, on generation accuracy (hybrid method only)**: `k=3` scored 48.5% (32/66), `k=5` scored 54.5% (36/66, the peak), `k=7` scored 47.0% (31/66) — notably below even the full-schema baseline, despite `k=7` having substantially higher retrieval recall (83.3%) than `k=5` (71.2%). This non-monotonic result was investigated directly by diffing the `k=5` and `k=7` predicted SQL for the five questions that regressed: only one of the five showed genuine table confusion (the same `lapTimes-vs-results` case reappearing in reverse, where `k=7`'s wider retrieved set reintroduced the distractor table that `k=5` had excluded); the other four showed logically equivalent SQL with stylistic differences only — extra type-casting, added `NULL` guards, or different clause placement — that diverged from the gold SQL's simpler formulation under the project's strict ordered-tuple grading. The conclusion drawn was that the `k=5-to-k=7` accuracy drop is driven mostly by the larger schema prompt subtly nudging the model toward more verbose, defensively-written SQL that mismatches gold's formulation under strict comparison, rather than primarily by genuine reasoning degradation from retrieving more (and noisier) tables — an important caveat on the `k-sweep`'s generation-accuracy numbers, distinguishing a grading-sensitivity artifact from a true retrieval-quality effect.

**Token cost comparison** Using a `cl100k_base` tokenizer as an approximate proxy (not Claude's exact tokenizer, but a reasonable stand-in for relative comparison), the full `formula_1` schema was measured at 2,539 tokens, constant across every question. The retrieved schema (`hybrid, k=5`) averaged 1,127 tokens per question — a 55.6% reduction, or roughly 1,412 tokens saved per question — achieved while simultaneously producing higher generation accuracy than the full-schema baseline.

## Deployment (Aided by Claude)

**Design choices**: the app would be a full showcase rather than a minimal demo, letting visitors toggle between four retrieval strategies (*full schema, BM25-only, embedding-only, and hybrid RRF*) with an adjustable `k` value, displayed alongside real eval statistics from the ablation work. The app would support both BIRD databases used across the project, `debit_card_specializing` and `formula_1`, selectable via a dropdown, with the understanding that they'd demonstrate different parts of the project: `debit_card_specializing` for the core agent and eval harness work, and `formula_1` for the retrieval work. Visitors would be able to type arbitrary free-text questions or use quick-fill buttons populated with curated example questions. Two architectural constraints shaped everything that followed:
* live `Voyage AI` and `Claude API` calls per click would be too slow and rate-limit-prone for a public-facing app, so we settled on a hybrid approach: precompute and bake in results for a curated set of example questions (so those are instant and free), while still supporting live API calls for free-text questions typed by visitors.
* retrieval indices (`BM25` corpus, `Voyage` embeddings) needed to be built once offline in Colab and shipped as static data files rather than rebuilt on every app startup, since rebuilding embeddings would mean unnecessary API costs and latency.

With that architecture settled, the work was sequenced into the following blocks:
1. *Offline export scripts*, run in the existing Colab notebook to generate data files for each database. For `debit_card_specializing`, this meant building, for the first time, its table-level schema chunks, a tokenized corpus for `BM25`, and `Voyage` embeddings for all five tables, then exporting four JSON files (`table names, table chunks, tokenized corpus`, and `embeddings`) to Google Drive. For `formula_1`, this was simpler since all of that infrastructure already existed in-session — the block just exported the existing `table_chunks, tokenized_corpus`, and `table_embeddings` variables directly, avoiding any redundant API calls.
2. *Curated example selection and precomputation*, wherein we picked three questions per database, chosen deliberately to tell a complete story rather than just show successes.
   * For `formula_1`, the three picks were question `850` (a clean win — full schema and hybrid retrieval both correct, with simple, legible SQL), question `904` (a case where retrieval fixed a mistake the full-schema baseline made, specifically correcting a column-confusion between the `results` and `lapTimes` tables), and question `954` (an honest failure — a case demonstrating the "join-bridge invisibility" failure mode identified before, where the results table got crowded out of retrieval and the model substituted an incorrect table).
   * For `debit_card_specializing`, since that database never had a retrieval ablation run against it, the three picks were chosen to show spread instead: question `1483` (a simple single-table lookup), question `1490` (a question requiring a join plus a percentage-formula hint from BIRD's evidence field), and question `1471` (a single-table question where the evidence field does the real work of defining what "ratio" means mathematically). 

For each curated question, we ran all four retrieval strategies through the existing `ask_db_with_retry function`, executed and graded each result against gold SQL, and saved everything — retrieved tables, generated SQL, predicted results, correctness, plus the gold SQL and gold result for comparison — into a precomputed JSON file per database. Since `debit_card_specializing` was decided to only ever expose "full schema" in the app's UI, only that one strategy was precomputed for its three questions, skipping retrieval infrastructure entirely for that database. This block also produced a `stats.json` file consolidating the project's key eval numbers: the `formula_1` strategy comparison (`full schema 50.0%, BM25-only 48.5%, embedding-only 53.0%, hybrid 54.5%, all at k=5`), the k-sweep results for both generation accuracy (`47.0% at k=3, 54.5% at k=5, 47.0% at k=7`) and retrieval recall (`53%, 71%, 83% respectively`), the token cost reduction at `k=5 (55.6%)`, and the `debit_card_specializing` baseline accuracy (`43.3%`).

3. *The Gradio app skeleton* — the UI layout and event wiring, using placeholder logic rather than real retrieval or generation calls, so the interface could be verified before connecting anything live. The layout included a database dropdown, a strategy dropdown and `k` slider (both conditionally hidden when `debit_card_specializing` is selected, since that database only supports full schema), a free-text question box, three quick-fill buttons per database that populate the textbox with curated questions, a run button, and an output area showing retrieved tables, generated SQL, the query result, and a gold-comparison section. A stats accordion was included to surface the eval numbers. This block was tested by loading dummy JSON files matching the real data schema and verifying that the Gradio Blocks graph compiled correctly, that switching databases correctly toggled the visibility of retrieval controls, and that quick-fill buttons populated with the correct per-database curated questions.
4. *Wiring in the real retrieval and generation logic*, broken into three sub-parts:
   * The first part set up the `Anthropic` and `Voyage` API clients (reading keys from environment variables, compatible with both Colab and Hugging Face Spaces' secrets mechanism), recreated the prompt templates and chains from the notebook, and added data-loading logic that builds a live `SQLDatabase` connection and rebuilds a `BM25Okapi` index in memory from the saved tokenized corpus for each database at app startup. This required a decision on how the app would obtain the actual BIRD SQLite database files — resolved by bundling them directly into the deployed repository rather than re-downloading them at startup.
   * The second part adapted the four schema-builder functions (full, BM25-only, embedding-only, and hybrid via Reciprocal Rank Fusion) from the notebook so they operate on a per-database dictionary of loaded data rather than closing over single-database global variables, since the app needed to support two databases simultaneously.
   * The third part wrote the real `run_query function`, which routes to the correct schema-builder based on the selected database and strategy, calls `ask_db_with_retry`, executes the result for display, and detects which tables actually made it into the generated schema text via a substring-matching trick. This function included layered error handling so that a failure at any stage (schema building, SQL generation, or execution) surfaces a readable message rather than crashing the app, and explicitly flagged that free-text questions lack BIRD's evidence hint field, meaning live free-text performance would likely run lower than the benchmarked numbers, which were evaluated with evidence included. This block was tested live, end-to-end, with a real free-text question about Michael Schumacher's fastest lap — which surfaced a genuinely interesting and previously undocumented failure mode: the hybrid retrieval strategy failed to retrieve the drivers table, yet the model still generated working SQL joining against it by correctly guessing the table's standard schema (column names like `forename` and `surname`) from general knowledge, rather than from anything actually shown in its context window.

5. *Wiring the curated-example short-circuit logic and a properly rendered stats panel*, wherein we decided that curated questions would always serve their precomputed `k=5` result regardless of the `k` slider's position, rather than falling through to a live call whenever `k` didn't match exactly, to preserve the cost and latency benefits of precomputation. The `run_query` function was extended to first check whether the typed question exactly matches one of the database's curated questions; if so, it returns the precomputed record for the selected strategy instantly, including a formatted gold-SQL and gold-result comparison with a correctness verdict, and only falls through to the live generation path for genuine free-text questions. The stats panel was rewritten from a raw JSON dump into readable markdown tables covering the strategy comparison, the k-sweep (both generation accuracy and retrieval recall), the token cost reduction, and the `debit_card_specializing` baseline. This was tested by submitting the same curated question under two different strategies and confirming both resolved instantly without live API calls.
